🎯 Why this baseline?

Traditional D365 F&O implementations rely on consultant memory and scattered spreadsheets. This workspace replaces all of that with a structured, AI-readable knowledge base where every capability is a Skill — loaded only when needed — and per-module work is delegated to sub-agents running in their own context windows.

🧠 Progressive disclosure

Each skill's description is the only always-loaded text. The full body loads on demand. The main agent's context stays small even on a 25-module project.

🤖 Sub-agent fan-out

Per-module work — config building, deployment, validation — runs in parallel sub-agents. Each loads only one module's knowledge. The orchestrator never sees module-level detail.

🔄 Self-improving

Every challenge is logged to the Challenge Journal and fed back into the relevant skill or module knowledge file. The next implementation gets a head start.

📋 Source-of-truth files

Configuration files in Modules/ and Documentation/ are authoritative. If the environment ever drifts, fix the source first, then re-deploy.

🔌 Live MCP integration

The fo-mcp-server skill teaches the agent the Dynamics 365 ERP MCP server — data tools, form tools, action tools. Built from the official Microsoft Learn article.

🧪 End-to-end testing

Tests follow complete business processes (Source-to-Pay, Order-to-Cash, Record-to-Report) start to finish — never fragments.

🏗️ Architecture

Four layers, 13 skills. The orchestrator at the top, leaf skills at the bottom; per-module work is fanned out to sub-agents in the middle.

flowchart TD U([User Request]) --> L0 subgraph L0["🟪 Layer 0 — Top-Level"] O1[phase-orchestrator] end L0 --> L1 subgraph L1["🟦 Layer 1 — Phase Skills"] P1[d365-requirements-analysis] P2[d365-config-builder] P3[d365-deployment] P4[d365-validation-testing] P5[d365-documentation] end L1 --> L2 subgraph L2["🟨 Layer 2 — Fan-Out"] F1[module-fanout] end L2 --> L3 subgraph L3["🟩 Layer 3 — Workers (Sub-Agents)"] W1[module-config-worker] W2[module-deployment-worker] W3[module-validation-worker] end L3 --> L4 subgraph L4["⬜ Layer 4 — Leaf Skills"] X1[d365-knowledge-routing] X2[fo-mcp-server] X3[reinforcement-learning] end L4 --> ENV[(D365 F&O Environment via MCP)]
Why aren't modules promoted to skills? Skills are capabilities; modules are reference knowledge. Promoting all 47 modules would add 47 always-loaded description blocks to every turn — pure context bloat. Instead, d365-knowledge-routing is the lookup index, and sub-agents load each module's knowledge only when actually needed.

Phase 1 — Analysis & Preparation

Think first, act later. No deployment until everything is documented.

  • 1.1 — Requirement profile + matrix
  • 1.2 — DMF templates + CSVs + parameter settings (per-module fan-out)
  • 1.3 — Rollout plan + E2E test plan
  • 1.4 — Validation report + gap resolutions
  • 1.5 — 9-point approval gate

Phase 2 — Deployment & Validation

Push via MCP. Test end-to-end. Fix-loop on failures.

  • 2.1 — Deployment in DMF dependency waves (per-module fan-out)
  • 2.2 — E2E scenarios start-to-finish (per-scenario-slice fan-out)

Phase 3 — Documentation

Seven single-file HTML deliverables + index.

  • 3.1 — rollout-report.html (narrative)
  • 3.2 — environment-config.html (visual reference)
  • 3.3 — test-results.html (E2E dashboard)
  • 3.4 — challenge-journal.html (journal browser + KPIs)
  • 3.5 — process-catalogue.html (APQC explorer)
  • 3.6 — dependency-graph.html (deployment waves)
  • 3.7 — run-state.html (deployment progress)
  • 3.8 — index.html (landing page)

🥉 Action tools

Priority 3 — X++ ICustomAPI classes

  • api_find_actions
  • api_invoke_action

Use for: Activate, Publish, Generate, Post, custom business logic.

Prerequisites
  • F&O version ≥ 10.0.47
  • Feature flag: "Dynamics 365 ERP Model Context Protocol server" enabled
  • Environment tier: Tier 2 or above, or a Unified Developer Environment (CHE not supported)
  • Agent platform added to Allowed MCP Clients form (defaults: Microsoft Copilot Studio, VS Code)
Cardinal MCP rules
  • Pre-flight every data op: data_find_entity_type → data_get_entity_metadata
  • Plural entity sets: SalesOrderHeaders, V2+ uses HeadersV2 not HeaderV2s
  • No deep insert — create parent, then children, in separate calls
  • Enum syntax: $filter=Style has Namespace.Pattern'Yellow'
  • Forms use internal names — never display labels
  • 25-row response cap — assume truncated, narrow the filter

🚀 Getting started

  1. Open the workspace in any Skills-aware AI host (Claude, VS Code Copilot Chat with Skills, etc.).
  2. Drop your requirement documents into Requirements/.
  3. Ask the agent: "Analyse the requirements" — phase-orchestrator classifies, then d365-requirements-analysis activates.
  4. Review Documentation/requirement-profile.md, iterate.
  5. "Build the configuration files" — d365-config-builder activates and fans out per-module sub-agents.
  6. Pass the 9-point approval gate, then "Deploy" — d365-deployment + fo-mcp-server activate.
  7. "Run end-to-end tests" — d365-validation-testing activates with the fix-loop.
  8. "Generate the rollout HTML" — d365-documentation produces the final deliverables.
Cardinal rules (apply across all skills):
  • Source files are authoritative — fix the source first, then re-deploy.
  • Document continuously. Never batch.
  • Follow processes start-to-finish. No fragment testing.
  • Consult module knowledge first when something is unexpected.
  • Feed back every discovery — every challenge becomes a future shortcut.
  • Respect DMF dependency order (010 → 650).
