Phase 1 Analysis & Preparation

Think first, act later. No deployment happens until every artefact is complete.

Step 1.1 — Analyse Requirements

Read all documents in Requirements/. Extract every functional requirement. Map each to D365 modules and classify as:

✅ Standard ConfigD365 supports out of the box
📦 Data MigrationRequires data loading via DMF
⚙️ Parameter SettingModule parameter toggle or value
🔄 WorkflowNeeds workflow or business rule
🔌 IntegrationExternal connection needed
⚠️ GapNo standard D365 capability
❓ ClarificationRequirement is ambiguous

Output: Documentation/requirement-profile.md + requirement-matrix.json

Step 1.2 — Build Configuration Files

For each in-scope module (in dependency order), create DMF template JSON, data CSV files, and parameter settings documents. Cross-verify posting profiles, dimension sets, number sequences, and tax setup across modules.

Output: Updated DMF templates, data files, config-{module}.md, parameter-settings.md

Step 1.3 — Build Rollout Plan + E2E Test Plan

Sequence all modules by DMF template number (010→650). For each entity, decide: data entity, form navigation, or API action. Build test scenarios that trace complete business processes from start to finish.

Output: rollout-plan.md, e2e-test-plan.md, test-scenarios.json, test-data-requirements.md

Step 1.4 — Validate & Fill Gaps

Cross-check every requirement against config files, deployment steps, and test scenarios. No requirement may be unaccounted for. Close all ❓ and ⚠️ items.

Output: validation-report.md, gap-resolutions.md

Step 1.5 — Approval Gate

9-point checklist verifying everything is complete: requirement profile, gaps resolved, config files exist, DMF templates sequenced, data files created, rollout plan covers all modules, E2E test plan covers all processes, validation report clean, Challenge Journal reviewed.

Phase 2 Deployment & Validation

Push configuration to D365 F&O via the MCP server, then validate with end-to-end testing.

Step 2.1 — Deploy via F&O MCP Server

For each module in sequence (010→650), execute 5 sub-steps:

  1. Pre-deployment check — verify upstream, read module knowledge, check Challenge Journal
  2. Deploy data entities — find entity type, get metadata, check existing data, create/update via OData
  3. Deploy form-based config — open forms, navigate tabs, set control values, save
  4. Execute actions — activate, publish, generate via API or form buttons
  5. Module validation — query entities, inspect forms, compare against config files
Step 2.2 — Validate + E2E Testing

Run every test scenario from start to finish. On failure: fix source files → re-deploy → re-test (the fix loop). Continue until all scenarios pass.

Verifications include: configuration spot-checks, posting profile verification, number sequence verification, and cross-module link verification.

Phase 3 Documentation

Generate two single-file HTML deliverables — narrative and reference.

Step 3.1 — rollout-report.html

Full narrative covering: project overview, requirements analysis, configuration design, deployment execution, testing & validation, challenges & learnings, appendices. Visual elements include timelines, pie charts, pass/fail matrices.

Step 3.2 — environment-config.html

Visual environment reference: org hierarchy, chart of accounts, site/warehouse structure, module reference cards, integration map, security model. Colour-coded by module area with interactive elements.

Data Tools — Key Rules

  • Get entity type + metadata before any CRUD
  • Entity names must be plural (SalesOrderHeaders)
  • V2+ entities: plural before V (HeadersV2)
  • No deep insert — parent then child
  • Enums: $filter=X has Namespace.Pattern'Value'

Form Tools — Key Rules

  • Use internal names, never display labels
  • Menu items, controls, tabs, columns — all by name
  • Never ask for menu item types
  • Max 25 rows per form query
  • Flow: find → open → click New → set values → save

Reasoning Protocol

  • Before tool call: plan + check knowledge + check journal
  • After tool call: reflect + check objective + next step
  • Never stop early, never substitute on failure
  • Never rely on general knowledge — always query live system

🚀 Getting Started

  1. Open the workspace in VS Code
    Open the ProcessBaseline folder with GitHub Copilot enabled. The master orchestrator (.github/copilot-instructions.md) loads automatically — no manual setup needed.
  2. Drop your requirement documents into Requirements/
    Place Word docs, PDFs, Excel files, or any text-based requirements into the Requirements/ folder. The agent will read all files in this folder during Phase 1.1 analysis.
  3. Ask Copilot to start Phase 1
    Tell the agent: "Analyse the requirements in the Requirements folder and build the requirement profile." The agent will classify, load the right module knowledge, and begin mapping requirements to D365 capabilities.
  4. Review and iterate through Phase 1
    The agent will produce configuration files, a rollout plan, and an E2E test plan. Review each artefact. Ask for changes. All work is saved to the Documentation/ folder continuously.
  5. Approve and deploy
    Once the Approval Gate passes, instruct: "Deploy the configuration following the rollout plan." The agent will push everything to D365 F&O via MCP tools, validate, and run E2E tests.
  6. Generate documentation
    After deployment validation: "Generate the rollout report and environment configuration HTML." Two single-file HTML deliverables are produced in Documentation/html/.

Key Principles to Remember

Source files are authoritative — if the environment drifts, fix the source first, then re-deploy.
Document continuously — write as you go in every phase, never batch documentation.
Follow processes start to finish — E2E testing must trace complete business processes, never fragments.
Consult module knowledge first — when something doesn't work, read the module .md file before trying fixes.
Feed back every discovery — every challenge, workaround, and rule goes back into the knowledge base.
Respect dependency order — always configure upstream modules before downstream consumers.
